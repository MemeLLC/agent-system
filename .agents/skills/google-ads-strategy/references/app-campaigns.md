# Google アプリキャンペーン

## 用語定義

| 略語 | 正式名称 | 定義 |
|------|---------|------|
| ACe | App Campaigns for engagement | 既存ユーザーの再エンゲージメント向けキャンペーン |
| ACi | App Campaigns for installs | 新規インストール獲得向けキャンペーン |
| ACpre | App Campaigns for pre-registration | リリース前の事前登録促進キャンペーン |
| MMP（AAP） | Mobile Measurement Partner | モバイル計測パートナー（AppsFlyer、Adjust、Singular等）。GoogleヘルプではAAP（App Attribution Partner）と表記されることがある |
| ATT | App Tracking Transparency | AppleのiOSトラッキング許諾フレームワーク |
| CPI | Cost Per Install | 1インストールあたりの広告費 |
| tCPI | Target CPI | 目標インストール単価 |
| tCPA | Target CPA | 目標アクション単価 |
| tROAS | Target ROAS | 目標広告費用対効果 |
| SKAN | SKAdNetwork | AppleのiOS向けプライバシー保護型アトリビューション |
| AAK | AdAttributionKit | SKANの後継となるAppleの新フレームワーク |
| SDK | Software Development Kit | アプリに組み込む開発キット（Firebase SDK等） |
| ディープリンク | Deep Link | アプリ内の特定画面に直接遷移させるリンク |

## キャンペーンタイプ別 早見表

| 目的 | キャンペーンタイプ | 入札戦略 | 日予算の目安 | 主な配信面 |
|------|-------------------|---------|------------|--------|
| **新規インストール獲得** | ACi（インストール） | tCPI | tCPI × 50以上 | Search, Play, YouTube, Display等 |
| **アプリ内アクション促進** | ACi（アクション最適化） | tCPA | tCPA × 10以上 | Search, Play, YouTube, Display等 |
| **収益最大化** | ACi（ROAS最適化） | tROAS | 十分な予算確保 | Search, Play, YouTube, Display等 |
| **既存ユーザー再エンゲージ** | ACe | tCPA | tCPA × 15以上 | Search, YouTube, Display等 |
| **リリース前の事前登録** | ACpre | 事前登録単価 | 事前登録単価 × 50以上 | Play, YouTube, Display等 |

> **注:** 配信面はアップデートにより変更・追加される可能性がある（Discover等）。

## 1. 配信面と基本特性

広告主がコントロールするのは「アセット（素材）」「予算」「入札目標」の3つ。他はすべてAIが自動で行う。

| 配信面 | 表示場所 | 特徴 |
|--------|---------|------|
| **Google検索** | 検索結果ページ | アプリ関連の検索クエリにテキスト広告として表示 |
| **Google Play** | ストア内検索結果・おすすめ | インストール意図の最も高いユーザーにリーチ |
| **YouTube** | インストリーム、インフィード、Shorts | 動画アセットで訴求。最大のリーチを提供 |
| **ディスプレイネットワーク** | アプリ内広告、モバイルWebサイト | 幅広いオーディエンスにリーチ |

## 2. 導入前提条件

| 項目 | 要件 |
|------|------|
| アプリストア掲載 | Google PlayまたはApp Storeに公開済み（ACpreを除く） |
| 計測SDK | Firebase SDK または MMP（AppsFlyer、Adjust等）を実装済み |
| コンバージョントラッキング | インストールまたはアプリ内イベントのトラッキング設定済み |
| 日予算 | tCPI × 50以上、またはtCPA × 10以上を確保可能 |
| ACe（エンゲージメント）追加要件 | 一定数のインストール（目安: 50,000以上）+ ディープリンク実装済み。要件はアカウントやカテゴリにより異なる場合があるため、[キャンペーン作成画面でACeが選択可能か確認](https://support.google.com/google-ads/answer/9234102)すること |

### 2-3. 導入前チェックリスト

- [ ] アプリストアの掲載情報が最適化されている（スクリーンショット、説明文、プレビュー動画）
- [ ] Firebase SDK または MMP が正しく実装されている
- [ ] Google Ads ↔ Firebase/MMP の連携完了。入札に使うPrimaryイベントをどちらから取るか決定済み
- [ ] インポートするコンバージョンの一覧（Primary / Secondary）を確定
- [ ] クリエイティブ最低セット: テキスト10本（見出し5+説明文5）、画像4枚以上、動画2本（16:9 + 9:16）
- [ ] 予算が入札 × 最低倍率を満たしている
- [ ] iOSの場合: SKANを前提にした判断窓（最低7日）をチームで合意
- [ ] ストア掲載（ASO）更新の担当・頻度を決定（アセット自動生成の品質に直結）
- [ ] ディープリンクの実装とテスト完了（ACeの場合）

## 3. キャンペーンタイプの選択

### 3-1. ACi（App Campaigns for installs）

新規インストールの獲得を目的とする最も基本的なタイプ。

**最適化目標の選択肢:**

| 最適化目標 | 入札戦略 | 推奨条件 |
|-----------|---------|---------|
| インストール数 | tCPI | CV数が少ない初期段階、まずボリュームを確保したい時 |
| アプリ内アクション | tCPA | 1日30件以上のターゲットイベントが見込める時 |
| アプリ内アクション値 | tROAS | 十分な購入/収益データがある時 |

**段階的な移行戦略:**

```
tCPI（インストール最適化）
  ↓ 1日30〜50件以上のアプリ内アクションが安定
tCPA（アクション最適化）
  ↓ 十分な購入・収益データが蓄積
tROAS（収益最適化）
```

### 3-2. ACe（App Campaigns for engagement）

既存ユーザーの再エンゲージメントを目的とするタイプ。

**利用条件（目安。アカウント・カテゴリにより変動する場合がある）:**
- 一定数のインストール基盤（目安: 50,000以上。以前は250,000だったが緩和傾向）
- ディープリンクの実装（App Links / Universal Links / カスタムスキーム）
- アプリユーザーのオーディエンスリスト

> **確認方法:** Google Adsのキャンペーン作成画面でACeが選択肢として表示されるかを確認するのが最も確実。[詳細](https://support.google.com/google-ads/answer/9234102)

**主な用途:**

| 用途 | 説明 |
|------|------|
| 休眠ユーザーの復帰 | 7日以上アプリを開いていないユーザーに広告配信 |
| カート放棄のリカバリ | 購入未完了ユーザーを商品ページにディープリンクで誘導 |
| アプリ内購入の促進 | 無課金ユーザーに課金コンテンツを訴求 |
| イベント・セールの告知 | 期間限定イベントのアプリ内ページに誘導 |

**ディープリンクの種類と優先度:**

| 種類 | プラットフォーム | 推奨度 |
|------|----------------|--------|
| App Links | Android | 最も推奨 |
| Universal Links | iOS | 最も推奨 |
| カスタムスキーム | iOS / Android | フォールバック用 |

### 3-3. ACpre（App Campaigns for pre-registration）

リリース前のアプリ・ゲームの事前登録を促進するタイプ。**Android限定。**

**前提条件:**
- Google Play Consoleでアプリの事前登録を有効化済み
- APKが少なくとも1つのリリーストラックにアップロード済み
- 事前登録開始後90日以内にリリース予定

**推奨タイミング:** リリース予定日の3〜6週間前に事前登録を開始し、同時にキャンペーンを開始。

**配信面:** Google Play、YouTube（インストリームのみ）、ディスプレイネットワーク。

**注意:** アプリがリリースされると、リリースされた地域のキャンペーンは自動的に配信停止される。

## 4. 計測基盤の構築

### 4-1. トラッキングソリューションの選択

| ソリューション | 対象OS | 概要 |
|--------------|--------|------|
| **Firebase SDK** | iOS / Android | Googleの公式モバイル開発プラットフォーム。Google Adsとの連携が最もスムーズ |
| **Google Playインストールリファラー** | Androidのみ | Google Play経由のインストール経路を軽量に計測 |
| **MMP** | iOS / Android | AppsFlyer、Adjust、Singular等。クロスプラットフォーム計測が強み |

**推奨:** Firebase SDKの導入を基本とし、必要に応じてMMPを併用する。Firebase SDKはGoogle Adsの入札最適化に必要なシグナル収集とオーディエンス構築に固有の機能を持つため、MMPと併用しても競合しない。

### 4-2. コンバージョンイベントの設計

**イベント選択の原則:** 量よりも質。インストール数だけでなく、ビジネスにとって価値のあるアプリ内イベントを最適化対象に選ぶ。

| イベント種別 | 例 | 用途 |
|------------|---|------|
| インストール | first_open | ACiの基本最適化対象。Androidは自動計測 |
| ライトイベント | チュートリアル完了、プロフィール登録 | tCPAの初期最適化対象（ボリューム確保用） |
| ミドルイベント | カート追加、レベルクリア、検索実行 | ユーザー品質が安定してきた段階の最適化対象 |
| ディープイベント | 購入、サブスクリプション開始、課金 | 最終的な最適化対象。十分なボリュームが必要 |

**目標イベント数の目安:**

| レベル | 1日あたりのイベント数 | 状態 |
|--------|---------------------|------|
| 最低ライン | 10件/日 | キャンペーンが学習可能な最小値 |
| 推奨ライン | 30〜50件/日 | 安定した最適化が可能 |
| 理想ライン | 100件以上/日 | 高精度な最適化。iOS SKANのプライバシー閾値にも有利 |

### 4-3. Firebase SDKの設定ポイント

1. **Firebase SDKをアプリに組み込む**
2. **Google AnalyticsをFirebaseプロジェクトで有効化する**
3. **カスタムイベントを定義する**（purchase、subscribe、level_complete等）
4. **FirebaseプロジェクトをGoogle Adsアカウントにリンクする**
5. **Google Adsでコンバージョンアクションとしてインポートする**

### 4-4. MMPとの併用

FirebaseとMMP（AppsFlyer、Adjust等）は併用可能で、以下のメリットがある:

| 役割 | Firebase | MMP |
|------|---------|-----|
| Google Ads入札シグナル | 最適 | 対応 |
| オーディエンス構築 | Firebase独自機能 | 非対応 |
| クロスプラットフォーム計測 | 限定的 | 主要機能 |
| 多媒体アトリビューション | Google中心 | 全媒体対応 |

### 4-5. 併用時の二重計上リスク

FirebaseとMMPを併用する場合、同一のコンバージョンイベント（例: purchase）を両方からGoogle Adsにインポートすると、二重計上が発生し入札最適化とレポートが崩れる。

**防止ルール:**

| ルール | 説明 |
|--------|------|
| Google Ads入札に使うPrimaryイベントは1系統に統一 | Firebaseからインポートするか、MMPからインポートするか、どちらかに決める |
| 同名イベントの重複インポートを避ける | Firebase経由のpurchaseとMMP経由のpurchaseを両方Primaryにしない |
| Primary / Secondary（補助）の使い分けを明確にする | 入札最適化に使うイベントはPrimary、レポート参照用はSecondaryに設定 |
| イベント命名規則を統一する | チーム内でFirebase/MMP間のイベント名と用途の対応表を作成・管理する |

## 5. キャンペーン構造の設計

### 5-1. 基本原則

アプリキャンペーンの構造設計で最も重要な原則: **統合を基本とし、明確な理由がある場合のみ分割する。**

過度な分割はデータの分散を招き、AIの学習効率を低下させる。

### 5-2. キャンペーンの分割基準

| 分割軸 | 理由 | 例 |
|--------|------|---|
| **OS別** | iOS / Androidで計測・最適化が根本的に異なる | iOS用 / Android用 |
| **最適化目標別** | tCPIとtCPAは同一キャンペーンで混在不可 | インストール用 / アクション用 |
| **地域別** | LTVやCPIが大きく異なる地域は分割 | 日本 / 東南アジア / 北米 |
| **言語別** | クリエイティブが異なる言語が必要な場合 | 日本語 / 英語 |

**注意:** 同じ地域で同じ最適化目標のキャンペーンを複数作らない。自社キャンペーン同士で競合し、CPIが上昇する。

**iOSの運用推奨:** iOSキャンペーンは原則8本以内を推奨。これはプラットフォームの上限ではなく、シグナル分散を防ぐための運用ガイドライン。ただし、地域・言語・最適化目標が大きく異なる場合は例外として分割を検討する。

### 5-3. 広告グループの設計

広告グループは「クリエイティブテーマ」単位で作成する。

| テーマ例 | アセット内容 |
|---------|------------|
| 機能訴求 | アプリの主要機能をハイライト |
| ユーザーベネフィット | ユーザーが得られる価値を訴求 |
| ソーシャルプルーフ | レビュー・評価・ダウンロード数 |
| プロモーション | 期間限定オファー・初回特典 |

**広告グループあたりのアセット上限:**

| アセット種別 | 上限数 | 推奨数 |
|------------|-------|--------|
| テキスト（見出し） | 5 | 5（すべて埋める） |
| テキスト（説明文） | 5 | 5（すべて埋める） |
| 画像 | 20 | 最低4枚以上 |
| 動画 | 20 | 最低2本以上（縦・横） |
| HTML5 | 20 | ゲームアプリのみ |

### 5-4. 推奨キャンペーン構造例

**基本構造（単一地域・Androidアプリ）:**

```
アカウント
└── ACi - JP - インストール
    ├── 広告グループ: 機能訴求
    ├── 広告グループ: ベネフィット訴求
    └── 広告グループ: ソーシャルプルーフ
```

**中規模構造（iOS + Android、複数地域）:**

```
アカウント
├── ACi - Android - JP - インストール（tCPI）
│   ├── 広告グループ: 機能訴求
│   └── 広告グループ: プロモーション
├── ACi - Android - JP - アクション（tCPA）
│   ├── 広告グループ: 課金訴求
│   └── 広告グループ: ベネフィット訴求
├── ACi - iOS - JP - インストール（tCPI）
│   └── 広告グループ: 機能訴求
├── ACe - Android - JP - リエンゲージ（tCPA）
│   └── 広告グループ: 休眠復帰
└── ACi - Android - SEA - インストール（tCPI）
    └── 広告グループ: 英語圏訴求
```

## 6. 入札戦略と予算設計

### 6-1. 入札戦略の選択

| 入札戦略 | 最適化対象 | 推奨条件 | 予算/入札比率 |
|---------|-----------|---------|-------------|
| **tCPI** | インストール数 | CV数が少ない初期段階 | 日予算 ≥ tCPI × 50 |
| **tCPA（新規インストール）** | アプリ内アクション | 1日30件以上のイベント | 日予算 ≥ tCPA × 10 |
| **tCPA（既存ユーザー・ACe）** | 再エンゲージアクション | 既存ユーザー基盤あり | 日予算 ≥ tCPA × 15 |
| **tROAS** | コンバージョン値 | 十分な購入データ | 十分な予算確保 |

### 6-2. 初期入札額の考え方

**tCPIの場合:**
- アプリカテゴリの平均CPIを調査（国・カテゴリで大きく異なる）
- 初期入札は平均CPIの80〜100%に設定
- 学習期間（1〜2週間）は高めのCPIを許容する

**tCPAの場合:**
- 実績ベースのCPA（Firebase/MMP計測値）を基準に設定
- 初期入札は実績CPAと同等か、やや高めに設定

### 6-3. 予算と入札の比率が重要な理由

アプリキャンペーンのAIは、予算と入札のバランスからシグナルを読み取る。比率が低すぎると:
- 1日の配信が少なすぎて学習データが不足する
- 「予算による制限（Budget limited）」ステータスになり、最適な配信ができない

| 入札戦略 | 最低比率 | 推奨比率 |
|---------|---------|---------|
| tCPI | 50倍 | 100倍 |
| tCPA（新規） | 10倍 | 20倍 |
| tCPA（ACe） | 15倍 | 20倍 |

### 6-4. 学習期間の運用

**学習期間:** 新規キャンペーン作成後、約1〜2週間。

**学習期間中のルール:**
- 入札額や予算を変更しない
- クリエイティブを大幅に変更しない
- ターゲティングを変更しない
- CPIやCPAが一時的に高くなっても許容する
- 2日以内のデータで判断しない（特にiOS）

**学習完了の判断基準:** 5〜7日間、CPAが安定していること。

### 6-5. tROAS運用時のコンバージョン値の定義

tROASを使用する場合、送信する `value`（コンバージョン値）の定義をチームで統一しておく必要がある。定義がブレると最適化が機能しない。

**明確にすべき項目:**

| 項目 | 確認事項 |
|------|---------|
| 収益イベントの範囲 | アプリ内購入、サブスクリプション、広告収益のうち、どれを含めるか |
| value / currency の送信ルール | 税込 or 税抜、割引適用後の金額、通貨コード |
| 返金の扱い | 返金時にマイナス値を送信するか、別イベントで管理するか |
| サブスクリプションの区別 | trial_start（無料体験開始）と paid_start（有料開始）を分け、最適化対象は原則「支払い発生」に寄せる。ボリューム不足時は段階的に |

## 7. テキストアセットの作成

### 7-1. テキストアセットの仕様

| 項目 | 上限 | 備考 |
|------|------|------|
| 見出し | 最大5個 | 各30文字以内 |
| 説明文 | 最大5個 | 各90文字以内 |

**重要:** 各テキストは独立したアセットとして扱われる。見出しだけで表示される場合、説明文だけで表示される場合、複数の組み合わせで表示される場合がある。すべてのテキストが単独でも意味が通るように書く。

### 7-2. テキスト作成のベストプラクティス

**やるべきこと:**

| ルール | 理由 |
|--------|------|
| 各テキストを独立した文として書く | テキストは単独でも組み合わせでも表示される |
| 長さを変える（短い/中程度/長い） | 配信面によって表示可能な文字数が異なる |
| USP（独自の強み）を明確に伝える | 競合との差別化ポイントが最重要 |
| 具体的な数字を入れる | 「100万DL突破」「4.8星評価」等は信頼性を高める |
| CTAを目的に合わせる | インストール→「今すぐダウンロード」、ゲーム→「プレイ開始」 |

**避けるべきこと:**

| NG | 理由 |
|----|------|
| アプリ名を見出しに入れる | アプリ名は広告フォーマットに自動表示される |
| 「インストールはこちら」等のCTA | CTAボタンは広告フォーマットに組み込み済み |
| すべて同じ長さのテキスト | 配信面ごとの最適化が効かなくなる |
| 見出しと説明文の内容重複 | 組み合わせ時に冗長な広告になる |

### 7-3. テキストアセットの例（フィットネスアプリの場合）

**見出し（各30文字以内）:**

| # | テキスト | 訴求軸 |
|---|---------|--------|
| 1 | 自宅で本格トレーニング | 機能 |
| 2 | 100万人が選んだフィットネス | ソーシャルプルーフ |
| 3 | 初回7日間無料で体験 | オファー |
| 4 | AIがあなた専用メニュー作成 | 差別化 |
| 5 | 運動習慣を今日からスタート | ベネフィット |

**説明文（各90文字以内）:**

| # | テキスト | 訴求軸 |
|---|---------|--------|

## 8. 画像アセットの作成

### 8-1. 画像アセットの仕様

| フォーマット | アスペクト比 | 推奨サイズ | ファイル形式 | 最大サイズ |
|------------|------------|----------|------------|----------|
| **横長** | 1.91:1 | 1200×628px | JPG / PNG | 1,024KB |
| **正方形** | 1:1 | 1200×1200px | JPG / PNG | 1,024KB |
| **縦長** | 2:3 | 1000×1500px | JPG / PNG | 1,024KB |

**優先度:** 1枚だけ用意する場合は**横長（1200×628px）**を優先。最も多くの配信面で使用される。理想は全フォーマットを用意すること。

### 8-2. 画像作成のベストプラクティス

| ルール | 説明 |
|--------|------|
| 画像全体を使い切る | 余白を極力減らし、フレームいっぱいに訴求 |
| テキストは控えめに（目安: 画像面積の20〜25%以下） | テキスト過多の画像はパフォーマンスが低下する傾向。Googleの厳密な制限ではないが、テキストが少ないほど視覚的インパクトが高い |
| ロゴやCTAを重ねない | 広告フォーマットに自動表示されるため不要 |
| 高解像度を使用する | ぼやけた画像はCTRを大きく低下させる |
| アプリUIのスクリーンショットを含める | 実際の使用感を伝え、インストール後の期待値を設定 |

**配信面別の注意:**

| 配信面 | 最適フォーマット | 注意点 |
|--------|----------------|--------|
| ネイティブ広告 | 横長 | 最も多く使われるフォーマット |
| インタースティシャル | 縦長 | 全画面表示。インパクト重視 |
| ディスプレイバナー | 正方形 | アプリ内バナー枠に表示 |

### 8-3. テストすべき画像バリエーション

| バリエーション | 内容 |
|--------------|------|
| 人物あり vs なし | ターゲット層に近い人物を使うとCTRが向上する場合がある |
| アプリUI vs ライフスタイル | 機能訴求 vs 利用シーン訴求 |
| テキスト入り vs ビジュアルのみ | 数値訴求（「4.8星」）vs 純粋なビジュアル |
| 明るい配色 vs ダーク配色 | ターゲット層やアプリのトーンに合わせる |

## 9. 動画アセットの作成

### 9-1. 動画アセットの仕様

| 項目 | 仕様 |
|------|------|
| アップロード先 | YouTubeに事前アップロードが必要 |
| 推奨長さ | 10〜30秒 |
| 最大数 | 広告グループあたり20本 |

**フォーマット別の用途:**

| フォーマット | アスペクト比 | 主な配信面 |
|------------|------------|----------|
| 横長 | 16:9 | YouTubeインストリーム |
| 縦長 | 9:16 | YouTube Shorts、アプリ内インタースティシャル |
| 正方形 | 1:1 | ディスプレイネットワーク、フィード面 |

### 9-2. 動画作成のベストプラクティス（ABCDフレームワーク）

Googleが推奨する動画広告のABCDフレームワーク:

| ステップ | 内容 | 実践方法 |
|---------|------|---------|
| **A**ttract（引きつける） | 冒頭で注意を引く | 最初の3秒でペインポイントを提示 or 驚きの要素 |
| **B**rand（ブランド） | ブランドを自然に紹介 | 最初の5秒以内にロゴまたはアプリアイコンを表示 |
| **C**onnect（つながる） | 感情的なつながりを構築 | ユーザーが共感できるストーリーやシーン |
| **D**irect（行動促進） | 明確なCTA | 「今すぐダウンロード」等の行動喚起で締める |

### 9-3. 動画の詳細ガイドライン

**強く推奨するルール:**

| ルール | 理由 |
|--------|------|
| 音声を入れる | 無音の広告はパフォーマンスが大幅に低下する傾向がある |
| アプリUIを早い段階で見せる | インストール後の体験を具体的にイメージさせる |
| テンポを速くする | 複数のカットでテンポよく進行。離脱を防止 |
| 小さい画面を想定する | モバイル視聴が前提。タイトなフレーミングと明るい映像 |
| 「ここをクリック」は使わない | 広告ポリシー違反のリスク |

**長さ別の使い分け:**

| 長さ | 配信面 | 用途 |
|------|--------|------|
| 10〜15秒 | Display, Play Store | 短い注意喚起。機能のハイライト |
| 15〜30秒 | YouTube, Display | メインの訴求動画。ABCD構成 |
| 30〜60秒 | YouTube Shorts（縦長） | Shorts面での配信。カジュアルなトーン |

### 9-4. 動画がない場合

動画アセットを登録しない場合、Googleがアプリストアの掲載情報（スクリーンショット等）から自動生成動画を作成する。ただし、自動生成動画は自作動画に比べてパフォーマンスが劣る傾向がある。

**強く推奨:** 横長（16:9）1本 + 縦長（9:16）1本を最低限用意する。動画がないとYouTube在庫の取り切りが難しく、自動生成動画への依存で品質が低下しやすい。

## 10. HTML5・プレイアブル広告

### 10-1. 概要

インタラクティブなプレイアブル広告。ユーザーがアプリの一部を広告内で体験できる。**主にゲームアプリ向け。**

### 10-2. 仕様

| 項目 | 仕様 |
|------|------|
| フォーマット | HTML5 |
| 最大数 | 広告グループあたり20個 |
| 音声・動画 | HTML5広告内では非対応 |
| 配信面 | ディスプレイネットワーク（アプリ内） |

### 10-3. 注意点

- HTML5広告は他のアセットタイプに比べてCPMが大幅に高い
- CPMの上昇はCPI / CPAにも影響する
- ゲームアプリ以外では費用対効果が合わない場合が多い
- 導入前にテスト期間を設け、ROIを慎重に評価すること

## 11. アセット運用とAd Strength

### 11-1. Ad Strength（広告の有効性）

各広告グループのAd Strengthを「良好（Good）」以上に維持する。

**Ad Strengthの評価基準:**
- アセットの種類と数が十分か
- アセットの多様性があるか（テーマ・フォーマットの違い）
- Googleのベストプラクティスに沿っているか

### 11-2. アセットパフォーマンスの評価

各アセットにはパフォーマンスレーティングが付与される:

| レーティング | 意味 | 対応 |
|------------|------|------|
| **学習中** | データ収集中 | 変更しない。最低数日〜1週間待つ |
| **低** | 同種アセットの中で最もパフォーマンスが低い | 新しいアセットに差し替え検討 |
| **良好** | 安定したパフォーマンス | 維持。類似バリエーションの追加を検討 |
| **最良** | 最もパフォーマンスが高い | この方向性で新しいバリエーションを作成 |

**評価の注意点:** レーティングだけで判断しない。以下のKPIを組み合わせて総合判断する。

| KPI | 確認ポイント |
|-----|------------|
| パフォーマンスレーティング | 学習中/低/良好/最良 |
| 費用配分（支出額） | 支出が少ないアセットはGoogleが自動生成アセットを優先している可能性がある |
| 配信量（インプレッション） | 十分な配信がないレーティングは信頼性が低い |
| CPI / CPA | アセット別のコスト効率 |
| CVR | アセット別のコンバージョン率 |

**判断の分岐:**
- 「最良」+ 高支出 + 許容CPI/CPA → この方向性で新バリエーションを作成
- 「低」+ 高支出 → 優先的に差し替え（コストインパクト大）
- 「低」+ 低支出 → 学習不足の可能性あり。もう少し様子を見るか差し替え
- 支出ゼロ → Googleが自動生成アセットを優先。アセットの品質・形式を見直す

### 11-3. アセット更新のルール

| ルール | 理由 |
|--------|------|
| 更新頻度は控えめに（目安: 週1〜2回） | 頻繁な変更はAIの学習をリセットする。学習が安定しているかを優先 |
| 全アセット一斉入れ替えはしない | 比較データがなくなる |
| 「最良」アセットの方向性で新規作成 | 勝ちパターンを横展開 |
| 「低」アセットから差し替える | 最もインパクトの大きい改善 |
| テーマの多様性を維持する | AIが多様な配信面・ユーザーに対応できる |

## 12. iOSキャンペーンの注意点

### 12-1. ATT（App Tracking Transparency）の影響

iOS 14.5以降、ATTプロンプトでユーザーの許諾が必要。一般的なオプトイン率は15〜30%程度と言われるが、アプリのカテゴリ・地域・プロンプト設計により大きく変動する。

**影響:**
- コンバージョンデータの大部分がモデリング（推定値）になる
- データの反映に最大5日の遅延が発生する場合がある
- キャンペーンの最適化精度がAndroidに比べて低下する

### 12-2. SKAdNetwork（SKAN）

AppleのプライバシーフレームワークでATTに同意しないユーザーのアトリビューションを集約レベルで計測。

**SKAN 4.0の主な特徴:**
- 最大3回のポストバック（0〜48時間、3〜7日、8〜35日）
- コンバージョン値: 0〜63の64段階
- ソース識別子: 最大4桁（10,000パターン）

**プライバシー閾値の課題:** 一定のインストールボリュームがないとコンバージョン値が「null」で返る。

| プラットフォーム | 推奨日次インストール数（目安） |
|----------------|---------------------|
| Google Ads | 100件以上/日 |
| Facebook | 88件以上/日 |

> **注:** これらの数値は各プラットフォームの推奨目安であり、Appleの閾値は非公開。カテゴリやキャンペーン構成により必要ボリュームは異なる。

### 12-3. iOS運用のベストプラクティス

| ルール | 理由 |
|--------|------|
| iOSキャンペーンは原則8本以内を推奨 | シグナル分散を防止（プラットフォーム上限ではなく運用ガイドライン） |
| 2日以内のデータで判断しない | モデリングデータの反映に時間がかかる |
| ATTプロンプトの最適化 | オプトイン率が高いほどデータ品質が向上 |
| Firebase SDKを必ず導入 | iOSのオンデバイス計測に必要 |
| 十分なインストールボリュームを確保 | SKAN閾値を超えるため |

### 12-4. iOS実装チェックリスト

| タスク | 説明 |
|--------|------|
| SKANコンバージョン値スキーマ設計 | どのアプリ内イベントをどのコンバージョン値（0〜63）に割り当てるかを設計。収益帯・エンゲージメント段階で分類 |
| ポストバック遅延を前提にした判断窓 | SKAN 4.0は最大35日の遅延。判断には最低7日の移動平均を使用 |
| ATTプロンプトの表示タイミングと文言最適化 | アプリの価値を体験した後（例: チュートリアル完了後）に表示するとオプトイン率が向上 |
| Firebase SDKのオンデバイス計測設定 | iOSのプライバシー要件に対応するために必須 |
| MMP側のSKAN対応確認 | MMP（AppsFlyer、Adjust等）のSKAN 4.0対応状況と設定手順を確認 |

### 12-5. AdAttributionKit（AAK）への移行

AppleはSKANの後継としてAdAttributionKit（AAK）を発表（WWDC 2025）。現時点でのMMP・広告プラットフォーム側の対応状況はまだ流動的なため、最新情報を定期的に確認すること。

## 13. 運用サイクルとスケーリング

### 13-1. 運用サイクル

| 頻度 | 確認項目 |
|------|---------|
| **毎日** | 予算消化ペース、CPI/CPA推移、異常値チェック |
| **週次** | アセットパフォーマンス確認、Ad Strength確認、アセット更新検討 |
| **隔週** | 入札額の微調整（±20%以内）、新規広告グループのテスト検討 |
| **月次** | キャンペーン全体のROI評価、構造の見直し、新規キャンペーンの検討 |

### 13-2. スケーリング戦略

**スケーリングの前提条件:**
- 5〜7日間、CPAが安定していること
- 1日あたり20〜30件以上のターゲットイベントがあること

**スケーリングのルール:**

| ルール | 詳細 |
|--------|------|
| 予算増加は48〜72時間ごとに最大20% | 急激な増加はAIの学習を不安定にする |
| 入札額の変更は24時間で最大20% | 大幅な変更はキャンペーンの学習をリセットする |
| 目標の変更時は新規キャンペーンを作成 | tCPI → tCPAへの切り替えは新規作成が推奨 |

### 13-3. YouTubeトラフィックの獲得

YouTubeは最大のリーチを提供する配信面だが、入札単価によってはYouTube面への配信が行われない場合がある。

**YouTube面へのリーチを増やす方法:**
- 高品質な動画アセット（横長 + 縦長）を必ず登録
- 入札額を段階的に引き上げてYouTube在庫へのアクセスをテスト
- 動画のABCDフレームワークを遵守

### 13-4. アプリストアの最適化（ASO）との連携

アプリキャンペーンはアプリストアの掲載情報をテキストやクリエイティブの自動生成に使用する。ストアの最適化はキャンペーンパフォーマンスに直結する。

| ASO要素 | 影響 |
|---------|------|
| アプリタイトル・説明文 | AIがキーワードシグナルとして使用 |
| スクリーンショット | 自動生成動画の素材に使用される |
| プレビュー動画 | ストアページのCVRに直結 |
| 評価・レビュー | ストアページの信頼性。低評価はCVRを大きく低下させる |
| ダウンロード数 | ソーシャルプルーフとして機能 |

## 14. Web to App Connect

### 14-1. 概要

Web to App Connectは、Webキャンペーン（Search、Shopping、PMax等）からアプリへのシームレスな遷移を実現する機能。Webキャンペーンからのアプリインストールやアプリ内アクションを計測できる。

### 14-2. 対応キャンペーンタイプ

| キャンペーンタイプ | 対応状況 |
|-------------------|---------|
| 検索キャンペーン | 対応 |
| ショッピングキャンペーン | 対応 |
| Performance Max | 対応 |
| YouTube広告 | 対応（2025年拡張） |
| Demand Gen | 対応（2025年拡張） |
| Hotel Ads | 対応（2025年拡張） |

### 14-3. 主な機能

| 機能 | 説明 |
|------|------|
| ディープリンク連携 | Web広告クリック → アプリの該当画面を直接開く |
| 統合コンバージョン | アプリイベントとWebイベントを自動統合 |
| Web to Appアクイジション | Webキャンペーンがアプリインストールに寄与した数を計測 |
| 統合オーバービューカード | WebとAppのパフォーマンスを並べて比較 |

### 14-4. 効果

Googleの発表によると、YouTube広告でWeb to App Connectを使用したブランドはコンバージョン率が2倍に向上したとの報告がある（業種・規模・地域により結果は異なる）。iOSでは特に有効で、Webキャンペーンのアプリインストール貢献度を初めて可視化できる。

## 15. よくある失敗パターンと回避策

### 15-1. キャンペーン設計の失敗

| 失敗パターン | 問題点 | 回避策 |
|------------|--------|--------|
| 同じ地域・同じ目標で複数キャンペーン | 自社キャンペーン同士が競合しCPIが上昇 | 統合を基本とし、分割は明確な理由がある場合のみ |
| 全アセットと全地域を1キャンペーンに詰め込む | 言語・LTV差が考慮されず最適化が機能しない | OS別・地域別・言語別で適切に分割 |
| iOSキャンペーンが多すぎる | シグナル分散で学習精度が低下 | 原則8本以内を推奨。地域・言語・目標が大きく異なる場合のみ例外的に分割 |

### 15-2. 予算・入札の失敗

| 失敗パターン | 問題点 | 回避策 |
|------------|--------|--------|
| 予算/入札比率が低すぎる | 「予算による制限（Budget limited）」で十分な学習ができない | tCPI × 50以上、tCPA × 10以上を確保 |
| 学習期間中に入札や予算を頻繁に変更 | AIの学習がリセットされ、最適化が進まない | 最低1〜2週間は変更しない |
| 一度に20%以上のスケーリング | 学習が不安定になりCPAが急上昇 | 48〜72時間ごとに最大20%ずつ |
| tCPIからtCPAへ同一キャンペーンで切り替え | 学習データのリセット | 新規キャンペーンを作成して切り替え |

### 15-3. クリエイティブの失敗

| 失敗パターン | 問題点 | 回避策 |
|------------|--------|--------|
| クリエイティブを長期間更新しない | クリエイティブ疲れでパフォーマンスが低下 | 定期的にアセットを更新（週1〜2回以内） |
| 全アセットを一斉に入れ替え | 比較基準がなくなり、何が効果的かわからない | 一部ずつ差し替え |
| 動画アセットを登録しない | YouTubeへの配信が制限される | 最低でも横長+縦長の2本を登録 |
| 画像にテキストを入れすぎる | 視認性低下、パフォーマンス低下 | テキストは控えめに（目安: 画像面積の20〜25%以下） |

### 15-4. 計測の失敗

| 失敗パターン | 問題点 | 回避策 |
|------------|--------|--------|
| コンバージョンイベントの選択が不適切 | ボリュームが少なすぎる or ビジネス価値が低い | 1日10件以上発生するイベントを選択。段階的に深いイベントに移行 |
| iOSで2日以内のデータで判断 | モデリングデータが反映されていない | 最低3〜5日のデータで判断 |
| Firebase/MMPの未導入 | アプリ内アクションの計測ができない | キャンペーン開始前に計測基盤を構築 |

### 15-5. 運用の失敗

| 失敗パターン | 問題点 | 回避策 |
|------------|--------|--------|
| アプリストアの掲載情報を放置 | 自動生成アセットの品質低下、CVR低下 | ストア情報を定期的に最適化（ASO） |
| 低評価のまま広告出稿 | ストアページでの離脱率が上昇 | 評価改善後にスケーリング |
| ローカライズなしで多言語地域に配信 | 言語が合わないユーザーにはほぼ効果なし | 主要言語別にキャンペーンを分割 |
