# Google Ads - GTM実装マニュアル

> 最新仕様は [Google Ads コンバージョン - GTMヘルプ](https://support.google.com/tagmanager/answer/6105160?hl=ja) で要確認。

## 1. コンバージョン設計

### プライマリ vs セカンダリ

| 分類 | 表示場所 | 入札最適化 | 用途 |
|------|---------|-----------|------|
| **プライマリ** | 「コンバージョン」列 | **対象** | ビジネスの最終目標（購入、リード獲得） |
| **セカンダリ** | 「すべてのコンバージョン」列のみ | 対象外 | 中間指標の観察（カート追加、ページ閲覧等） |

- 入札最適化に使われるかは「キャンペーンがどのコンバージョン目標を使っているか」に依存する。セカンダリCVでも、カスタム目標に追加してキャンペーンに紐付ければ入札対象になる
- プライマリ/セカンダリの設定はアカウント単位で管理

### カウント方法

| 設定 | 動作 | 推奨シーン |
|------|------|-----------|
| **すべて（Every）** | 同一ユーザーの複数回をすべてカウント | EC購入（各購入が売上に直結） |
| **1回（One）** | 同一ユーザーは最初の1回のみ | リード獲得（2回目の送信は追加価値が低い） |

### コンバージョンウィンドウ

| ウィンドウ | 説明 | デフォルト | 設定範囲 |
|-----------|------|----------|---------|
| クリックスルー | 広告クリック後のCV計測期間 | 30日 | 1〜90日 |
| ビュースルー | 広告閲覧（クリックなし）後のCV計測期間 | 1日 | 1〜30日 |
| エンゲージビュー | 動画広告専用（10秒以上視聴等） | 3日 | 1〜30日 |

### コンバージョン値

| 方式 | 説明 | 用途 |
|------|------|------|
| **静的値（固定値）** | 全CVに同一の値 | リードフォーム送信 = 5,000円 等 |
| **動的値** | トランザクション固有の値をdataLayerから取得 | EC購入（注文金額） |

**コンバージョン値ルール**: オーディエンス・地域・デバイスに基づいてCV値を自動調整可能（例: 高価値顧客リストのCV値を1.5倍に）。

### アトリビューションモデル

| モデル | 説明 |
|--------|------|
| **データドリブン（DDA）** | 機械学習で各タッチポイントの貢献度を配分（**デフォルト・推奨**） |
| **ラストクリック** | 最後のクリックに100%。2023年以前作成の既存CVアクションではこちらがデフォルトの場合あり |

> ファーストクリック、線形、減衰、接点ベースは2023年に廃止済み。

## 2. GTM構成

### 前提

- Google Adsで**コンバージョンID**（`AW-XXXXXXXXX`）と各**コンバージョンラベル**を取得済み
- gtag.jsとGTMで**同一目的のタグを重複実装しない**（GTM実装と直書きが混在して二重送信になる構成を禁止）

### タグ一覧

| タグ名 | タグタイプ | トリガー | 必須同意 |
|--------|----------|---------|---------|
| Google Ads - Conversion Linker | コンバージョン リンカー | All Pages | ad_storage |
| Google Ads - Remarketing | Google 広告のリマーケティング | All Pages | ad_storage, ad_personalization |
| Google Ads - CV Purchase | Google 広告のコンバージョン トラッキング | CE - purchase | ad_storage |
| Google Ads - CV Lead | Google 広告のコンバージョン トラッキング | CE - generate_lead | ad_storage |
| Google Ads - EC Data (Purchase) | Google Ads User-Provided Data Event | CE - purchase | ad_storage, ad_user_data |

### コンバージョンリンカー

広告クリック情報（GCLID）をURLから検出し、ファーストパーティCookieに保存するタグ。**原則、All Pagesで設置する**（互換性・保険のため）。ただし、GTM内でGoogleタグ（GA4設定タグ等）を全ページで読み込んでいる構成では、Googleタグがこの機能を内包するため不要な場合がある。

**保存されるCookie:**

| Cookie | 用途 |
|--------|------|
| `_gcl_aw` | Google Ads クリックID |
| `_gcl_dc` | DoubleClick クリックID |
| `_gcl_gs` | Google Ads セッション情報 |

**設定:**
1. タグタイプ:「コンバージョン リンカー」
2. トリガー: **All Pages**（必須）

**クロスドメイン設定:**
- 「ドメイン間でのリンクの有効化」にチェック
- 自動リンクドメインをカンマ区切りで入力
- 必要に応じて「フォーム内でもリンクを有効化する」にチェック

### コンバージョンタグ

**タグ設定:**
1. タグタイプ:「Google 広告のコンバージョン トラッキング」
2. コンバージョンID・ラベルを入力
3. コンバージョン値・通貨コード・トランザクションIDをデータレイヤー変数で設定

### トリガーパターン

| パターン | トリガータイプ | 条件例 | 備考 |
|---------|--------------|--------|------|
| サンクスページ | ページビュー | Page URL が `/thank-you` を含む | 最もシンプル |
| フォーム送信 | フォームの送信 | Form ID が `contact-form` と等しい | 「タグの配信を待つ」「妥当性をチェック」推奨 |
| **dataLayerイベント（推奨）** | カスタムイベント | イベント名 = `purchase` 等 | 最も柔軟で信頼性が高い |

### 変数

**定数変数**（ID/ラベルを一元管理）:

| 変数名 | 値 |
|--------|------|
| `Google Ads - Conversion ID` | （コンバージョンID） |
| `Google Ads - CV Label Purchase` | （コンバージョンラベル） |
| `Google Ads - CV Label Lead` | （コンバージョンラベル） |

**データレイヤー変数**:

| 変数名 | データレイヤー変数名 |
|--------|-------------------|
| `DLV - ecommerce.value` | `ecommerce.value` |
| `DLV - ecommerce.currency` | `ecommerce.currency` |
| `DLV - ecommerce.transaction_id` | `ecommerce.transaction_id` |
| `DLV - ecommerce.items` | `ecommerce.items` |
| `DLV - enhanced_conversion_data` | `enhanced_conversion_data` |

### dataLayer実装例（購入完了）

```javascript
dataLayer.push({ ecommerce: null });
dataLayer.push({
  'event': 'purchase',
  'ecommerce': {
    'transaction_id': 'T-20260221-001',
    'value': 15000,
    'currency': 'JPY',
    'items': [
      { 'item_id': 'SKU-001', 'item_name': '商品A', 'price': 5000, 'quantity': 2 },
      { 'item_id': 'SKU-002', 'item_name': '商品B', 'price': 5000, 'quantity': 1 }
    ]
  },
  'enhanced_conversion_data': {
    'email': 'user@example.com',
    'phone_number': '+819012345678'
  }
});
```

> `ecommerce: null` の事前pushは必須（前回のecommerceデータとの混在を防止）。

## 3. 拡張コンバージョン（Enhanced Conversions）

ユーザー提供のファーストパーティデータ（メール等）をGoogleアカウントとマッチングし、Cookie非依存のCV計測を補完する機能。

### 2つのタイプ

| タイプ | 対象 | 用途 |
|--------|------|------|
| **Web向け** | オンラインCV | ウェブサイト上で完結するCVの精度向上 |
| **リード向け** | オフラインCV | リードフォーム送信後のオフラインCV追跡 |

### 送信可能なデータフィールド

| フィールド | 備考 |
|---|---|
| メールアドレス | **最もマッチ率が高い**。小文字に正規化 |
| 電話番号 | E.164形式（+81XXXXXXXXXX） |
| 名（ファーストネーム） | 小文字に正規化 |
| 姓（ラストネーム） | 小文字に正規化 |
| 住所（番地）/ 市区町村 / 地域 / 郵便番号 / 国コード | ISO 3166-1 alpha-2 |

> いずれか1つ以上必須。複数送信でマッチ率向上。

### GTMでの設定

**前提:** Google Ads > 目標 > コンバージョン > 設定 > 拡張コンバージョンをオン > 方法:「GTM」を選択

**方法A: Google Ads User-Provided Data Event タグ（推奨）**
1. タグタイプ:「Google Ads User-Provided Data Event」
2. コンバージョンIDを入力
3. User-provided dataセクションでデータレイヤー変数を設定
4. トリガー: コンバージョンタグと同じトリガー

**方法B: 既存コンバージョンタグに追加**
1. 既存CVタグの「拡張コンバージョン データを含める」にチェック
2. ユーザー提供データの変数を設定

### ハッシュ化

| パターン | 渡すデータ | フィールド名 | ハッシュ化の主体 |
|----------|-----------|-------------|----------------|
| **A: タグに任せる（推奨）** | 平文 | `email`, `phone_number` | gtag.js / GTMが自動ハッシュ |
| **B: 自前でハッシュ** | SHA-256ハッシュ値 | `sha256_email_address`, `sha256_phone_number` | 実装者が事前にハッシュ |

> **二重ハッシュ禁止:** パターンAのフィールド名にハッシュ値を渡す、またはパターンBのフィールドをタグが再ハッシュすると、マッチング不能になる。

**自前ハッシュ時の正規化ルール（パターンBの場合のみ）:**
1. trim（前後の空白削除）
2. toLowerCase（小文字変換）
3. 電話: E.164形式に統一
4. メール: gmail.comのピリオド削除・`+`以降の削除は必要な場合のみ（要検証。公式が必須とするのはtrim/小文字化まで）

### PII取り扱い注意

ユーザーデータ送信の推奨順位:

1. **サーバーサイド（sGTM等）でハッシュ→送信**（ページに平文を出さない。最も安全）
2. **Web側で送信する場合は、タグの自動ハッシュに任せる**（dataLayer経由でGoogle Adsタグが自動SHA-256化。Google公式の正規手順）
3. 自前ハッシュは正規化ミスのリスクがあるため、上記で対応できない場合のみ

dataLayerに投入されたPIIは、GTMコンテナ内の**すべてのタグ**（GA4、Meta Pixel等）からアクセス可能。対策:
- 拡張CV用の変数はGoogle Ads関連タグのみで使用する設計にする
- GTMプレビューモードや監視ツールにPIIが表示される可能性を認識する
- 同意取得後に限定してデータを送信する設計にする

### 拡張コンバージョン診断指標

| 指標 | 説明 |
|------|------|
| カバレッジ | ユーザー提供データが含まれるイベントの割合。100%に近づける |
| マッチ率 | Googleログインデータとマッチした割合 |
| アップリフト | 拡張CVにより追加で記録されたCV増加率 |

## 4. GA4連携

### リンク設定

GA4 管理画面 > プロダクトリンク > Google 広告のリンク > アカウント選択 > パーソナライズド広告を有効化。

| プラットフォーム | 必要な権限 |
|---|---|
| GA4 | プロパティレベルの「編集者」以上 |
| Google Ads | アカウントの「管理者」権限 |

### リンクで可能になること

| 機能 | 方向 | 説明 |
|------|------|------|
| キャンペーンデータ確認 | Ads → GA4 | クリック数、コスト、インプレッション |
| キーイベントのインポート | GA4 → Ads | GA4のキーイベントをCVとして利用 |
| オーディエンス共有 | GA4 → Ads | GA4オーディエンスを広告ターゲティングに |
| 自動入札の最適化 | GA4 → Ads | GA4のCVデータでSmart Biddingを最適化 |

### ネイティブCV vs GA4インポートCV

| 比較項目 | ネイティブ（推奨） | GA4インポート |
|----------|------------------|--------------|
| ビュースルーCV | 計測可能 | **不可** |
| クロスデバイスCV | 計測可能 | **制限あり** |
| データ反映速度 | 数時間以内 | 24〜48時間 |
| 拡張コンバージョン | フル対応 | 制限あり |
| 入札最適化 | 最適 | ビュースルー・クロスデバイスCV欠如で不利 |
| アトリビューション範囲 | Google Ads内のみ | 全チャネル横断 |

> GA4インポートが適するケース: GA4を計測の主軸にしている場合、Web以外（アプリ等）が対象の場合、ネイティブタグ設置が困難な場合。

### 推奨構成: ハイブリッドアプローチ

```
【プライマリ（入札最適化に使用）】
→ Google Ads ネイティブCVタグ（GTM経由）
  ・拡張コンバージョン有効
  ・ビュースルー・クロスデバイス対応

【セカンダリ（観察・分析用）】
→ GA4 キーイベントインポート
  ・全チャネル横断の分析に活用
  ・入札には使用しない（二重カウント防止）

※ 同じイベントを両方「プライマリ」にしない
```

### 数値の乖離

Google AdsとGA4間で20〜30%程度の数値差が発生しうる（目安。商材・設定により異なる）。

主な原因:
- アトリビューションモデルの違い
- 基準日の違い（GA4=CV発生日、Ads=クリック日）
- ビュースルーCV（Adsのみカウント可能）
- データ処理タイミング差、無効トラフィックフィルタリング基準の違い

### GA4オーディエンス活用例

| オーディエンス名 | 条件例 | 活用 |
|---|---|---|
| カート放棄者 | add_to_cart + purchaseなし（7日間） | リマーケティング |
| 高エンゲージメント | session_duration > 180秒 + page_view >= 5 | 類似オーディエンス |
| 既存顧客 | purchase発生（365日） | 除外設定、アップセル |

**予測オーディエンス:** 7日以内の購入見込み、28日間の高額支出見込み、7日以内の離脱見込み（利用条件: 正例・負例それぞれ1,000件以上/28日間）。

## 5. 動的リマーケティング

ユーザーが閲覧した商品情報を送信し、パーソナライズされた広告を表示する機能。

### GTM設定

1. リマーケティングタグの「動的リマーケティング イベントのデータを送信」にチェック
2. イベント名: データレイヤー変数 `{{Event}}`
3. イベント値・アイテム: 対応するデータレイヤー変数
4. トリガー: `view_item` / `add_to_cart` / `purchase` のカスタムイベント

### dataLayer実装例

```javascript
// 商品詳細ページ
dataLayer.push({
  'event': 'view_item',
  'value': 5000,
  'items': [{ 'id': 'SKU-001', 'google_business_vertical': 'retail' }]
});
```

> 動的リマーケティングの `id` は **Google Merchant Center フィードの商品IDと完全一致**させる必要がある。

## 6. Consent Mode

### Google Adsで重要な同意タイプ

| Consent タイプ | 対象 |
|---|---|
| `ad_storage` | 広告Cookie |
| `ad_user_data` | 広告目的のユーザーデータ送信 |
| `ad_personalization` | リマーケティング等の広告パーソナライゼーション |

### Basic vs Advanced

| モード | 動作 | データ収集 |
|---|---|---|
| **Basic** | 同意が得られるまで全タグをブロック | 同意拒否時はデータ収集なし |
| **Advanced（推奨）** | 同意状態に関係なくタグは読み込み、Cookieなし Pingを送信 | 同意拒否時もCookieless Pingを送信。モデリング可能 |

### GTM発火順序

1. `Consent Initialization - All Pages`（同意デフォルト設定）
2. `Initialization - All Pages`（初期化タグ）
3. `All Pages`（通常のページビュータグ）

### コンバージョンモデリング

Advanced Consent Mode有効時、Googleは同意拒否ユーザーの匿名Cookieless Pingと同意済みユーザーの行動パターンを比較し、AIでCVを推定。「モデル化されたコンバージョン」としてレポートに含まれる。

### 日本での扱い

Googleのプラットフォーム要件としては義務化されていないが、実装推奨（データ品質向上、グローバル展開への備え、改正個人情報保護法への対応）。

## 7. サーバーサイドGTM

### 概要

従来ブラウザで実行されていたタグ処理をサーバー上で実行する仕組み。

```
【クライアントサイド】 ブラウザ → 各プラットフォームサーバーに直接通信
【サーバーサイド】   ブラウザ → 自社サーバーコンテナ → 各プラットフォーム
```

| 利点 | 説明 |
|---|---|
| パフォーマンス向上 | クライアントサイドのJS実行量が減少 |
| データ品質向上 | 広告ブロッカーの影響を**軽減**（完全回避ではない） |
| Cookie耐久性 | HttpOnly Cookie設定可能（ITPの影響を軽減） |
| データ制御 | サーバー上でデータ加工・フィルタリング |
| セキュリティ | APIキー等をサーバー側で管理 |

### 設定概要

1. GTM > コンテナ作成 > Server を選択
2. Cloud Runへデプロイ
3. DNS CNAME: `gtm.yourdomain.com` → サーバーコンテナURL
4. クライアントサイドGA4設定タグで `transport_url: https://gtm.yourdomain.com` を指定
5. サーバーコンテナ側にGoogle Ads CVタグを設定

> sGTMには2つの経路がある: **(A)** ブラウザ→sGTM→GA4（GA4経由でAdsへ配送）と **(B)** sGTMで直接Ads CVタグを発火。上記手順は(A)の構成。(B)の詳細は [Google Ads CV（サーバーサイド）ガイド](https://developers.google.com/tag-platform/tag-manager/server-side/ads-setup) を参照。

### 導入判断基準

- 広告費が月額100万円以上
- Safari/Firefoxユーザーの割合が高い
- Cookie同意拒否率が高い市場（欧州等）

## 8. オフラインコンバージョン

### 概要

オンラインの広告クリックがオフラインの成果（商談成約、店頭購入等）につながった場合に、そのデータをGoogle Adsにインポートする仕組み。

### GCLID

広告クリック時に自動生成されるユニーク識別子。URLパラメータとして付加 → ファーストパーティCookie（`_gcl_aw`、有効期限90日）に保存。ランディング時にサーバーサイドで保存し、CV発生時にインポートする。

**関連識別子:**

| 識別子 | 用途 |
|--------|------|
| GCLID | 通常のWebクリックトラッキング |
| GBRAID | iOSアプリコンバージョン用 |
| WBRAID | iOS Webコンバージョン用 |

### インポート方法

| 方法 | 説明 |
|------|------|
| Google Ads管理画面 | 目標 > コンバージョン > アップロード > CSV |
| **Google Ads Data Manager（推奨）** | データソース接続 > スケジュール自動インポート |
| Google Ads API | プログラマティックなアップロード |

### インポートCSV形式

| フィールド | 形式 |
|-----------|------|
| Google Click ID | GCLID値（大文字小文字区別あり） |
| Conversion Name | コンバージョンアクション名（完全一致） |
| Conversion Time | `yyyy-mm-dd HH:mm:ss+\|-HH:mm` |
| Conversion Value | 数値（任意） |
| Conversion Currency | ISO 4217（任意） |

### 制限

- GCLIDの有効期限: クリックから**90日以内**にインポート必要
- 同じGCLID + コンバージョン名 + 日時の組み合わせは1回のみ
- 反映時間: 最大3時間

### リード向け拡張コンバージョン

従来のOCIの改良版。GCLIDに加えてハッシュ化ユーザーデータでもマッチング可能（中央値で約10%のCV増加）。

## 9. 自動入札とコンバージョンデータ

### 入札戦略と推奨CV数

| 入札戦略 | 最適化目標 | 推奨CV数/30日（目安） |
|----------|-----------|---------------------|
| CV数最大化 | CV数 | 15件以上 |
| 目標CPA | 目標コスト/CV | 30件以上（理想50+） |
| CV値最大化 | 合計CV値 | 15件以上 + 値の設定 |
| 目標ROAS | 広告費用対効果 | 50件以上 + 正確な値 |

> 推奨CV数は目安。商材・業種・CV単価により変動する。

### マイクロCV vs マクロCV

- **マクロCV**: ビジネスの主要目標（購入、リード獲得）→ 原則プライマリ
- **マイクロCV**: 途中のアクション（カート追加、フォーム入力開始）→ 原則セカンダリ

**マイクロCVをプライマリに使うケース:**
- マクロCVが月50件未満でSmart Biddingの学習に不十分
- B2B/高額商品でCVサイクルが数週間〜数ヶ月
- 新規キャンペーンのコールドスタート期間

**マイクロCVの値の算出:** マクロCVの平均値 × 遷移率（例: 購入10,000円 × checkout→購入40% = begin_checkout 4,000円）

### 推奨設定パターン（ECサイト例）

```
プライマリコンバージョン:
├── purchase（購入） → 値: 動的（注文金額）
└── lead_form_submit → 値: 固定（推定LTV）

セカンダリコンバージョン:
├── add_to_cart
├── begin_checkout
└── GA4インポート purchase

キャンペーン設定:
├── P-MAX（EC） → CV目標: purchase / 入札: tROAS
├── 検索（ブランド） → CV目標: purchase / 入札: CV数最大化
└── 検索（リード） → CV目標: lead_form_submit / 入札: tCPA
```

### データ品質向上施策

- 拡張コンバージョン有効化
- コンバージョン値を正確に設定（動的な値が理想）
- ネイティブタグを使用してデータ反映速度を向上
- 一貫した計測設定を維持

## 10. デバッグ

| ツール | 確認内容 |
|--------|---------|
| **GTMプレビューモード** | タグ発火順序、dataLayerの状態、変数の値、同意状態 |
| **Tag Assistant** | タグの検出・診断。tagassistant.google.comでデバッグセッション共有 |
| **DevTools Network** | `googleads.g.doubleclick.net/pagead/conversion/` へのリクエスト（cv=ラベル、value=値、oid=トランザクションID、gclaw=GCLID） |

### データ反映時間

| 条件 | 反映時間（目安） |
|------|---------|
| ラストクリック | 最大3時間 |
| DDA含むその他 | 最大15時間 |
| オフラインCVインポート | 最大3時間 |

> アカウント状況や処理遅延で上記を超える場合がある。直近データが少なく見える場合はCVの遅延（クリック日帰属）も考慮すること。

### よくあるトラブル

| 症状 | 確認事項 |
|------|---------|
| CVが記録されない | コンバージョンリンカーがAll Pagesで発火しているか / ID・ラベルが正しいか / トリガー条件 / GTMが公開済みか / Consent Modeでブロックされていないか |
| タグが「Failed」 | 広告ブロッカーを無効化してテスト / Consentタブで同意状態確認 / Networkタブでリクエスト送信を確認（偽陽性の可能性） |
| CV値が0または空 | dataLayerのpushタイミングがタグ発火より前か / DLV変数名が正しいか（大文字小文字、ドット記法） |
| GCLIDが保存されない | コンバージョンリンカーがAll Pagesか / リダイレクトチェーンでGCLIDが失われていないか / クロスドメイン設定 |
| 拡張CVのマッチ率が低い | メールアドレスを最優先で送信 / 複数データポイント送信 / 正規化の確認 / 自動収集から手動設定に切替 |
| 重複CV | `transaction_id` で重複排除（**同一コンバージョンアクション内**で有効）。SPA・決済戻りでの二重発火はsessionStorage等で発火ガードも検討 |

## 11. 実装チェックリスト

### 基本設定

- [ ] GTMコンテナが全ページに設置
- [ ] gtag.jsとGTMの二重実装がない
- [ ] GA4とGoogle Adsがリンク済み
- [ ] Google Adsの自動タグ設定が有効

### コンバージョン設定

- [ ] コンバージョンリンカーが「All Pages」で設定・公開
- [ ] ネイティブCVタグがGTMで設定
- [ ] カウント方法が適切（購入=すべて、リード=1回）
- [ ] ウィンドウが商材の検討期間に合致
- [ ] アトリビューションモデルがDDA
- [ ] プライマリ/セカンダリの分類が適切
- [ ] コンバージョン値が設定済み
- [ ] 同じイベントのプライマリが二重になっていない
- [ ] GA4キーイベントがセカンダリとしてインポート済み

### 拡張コンバージョン

- [ ] Google Adsで拡張CVが有効化
- [ ] User-Provided Data EventタグまたはCVタグの拡張CV設定が完了
- [ ] メールアドレスが最低限送信されている
- [ ] 正規化（小文字変換、空白除去）が正しい
- [ ] プレビューモードでユーザーデータの取得を確認

### Consent Mode

- [ ] Consent Mode v2が実装
- [ ] CMPが導入済み
- [ ] デフォルトの同意状態が適切
- [ ] Advanced Consent Modeが有効（Cookieless Ping送信）
- [ ] 同意拒否時の挙動をテスト済み

### 動的リマーケティング

- [ ] リマーケティングタグが設定
- [ ] 動的パラメータが送信されている
- [ ] 商品IDがMerchant Centerフィードと一致

### 継続運用

- [ ] CVステータスが「記録中」であることを定期確認
- [ ] Google AdsとGA4のCV数の乖離を定期チェック
- [ ] 拡張CVのマッチ率を確認
- [ ] 自動入札の学習状態を確認
- [ ] GTMのバージョン管理が適切
